# Coding Conventions

## CLI Builder Generated Project Conventions

All CLI projects generated by cli-builder follow:

- TypeScript ESM only
- Single quotes, 4-space indent, 100 char line width
- Commander + Zod + Chalk 5.x + tsup + Vitest + @clack/prompts
- pnpm workspace + ESLint + Sheriff + Prettier + Husky + Changesets
- Barrel exports enforced
- `.spec.ts` test naming

These conventions are defined in `plugins/cli-builder/skills/cli-create/references/tech-stack.md`.

**Simple mode**: For lightweight projects, use `simple: true` flag to skip complex tooling setup (ESLint, Sheriff, Changesets).

## Code Style Plugin Conventions

The `code-style` plugin enforces style rules during AI code generation and editing.

### Naming

| Element | Convention | Example |
|---------|-----------|---------|
| Variables/functions | `camelCase` | `getUserData` |
| Components | `PascalCase` | `UserProfile` |
| Constants | `UPPER_SNAKE_CASE` | `MAX_RETRIES` |
| Booleans | `is`/`has`/`should` prefix | `isValid` |
| Event callbacks | `handle`/`on` prefix | `handleClick` |
| Files/folders | `kebab-case` | `user-profile.tsx` |

### Import Order

1. Core framework/library imports
2. Third-party imports
3. Internal modules (nearest to farthest)
4. Type imports
5. CSS/SCSS imports (last)

### File Size Limits

- Maximum 800 lines per file
- Functions under 50 lines
- Extract utilities and split modules proactively

### Comments

- Public APIs and complex logic: JSDoc required
- Components: rely on TypeScript types, no redundant comments
- Inline: important code blocks need context comments
- Tags: `TODO`, `FIXME`, `HACK` with descriptions

### Error Handling

- Top-level unified handling with local fallback
- Web projects: native `Error`
- CLI/tool projects: custom Error types
- Messages: error code + readable message + context + stack
- React/Vue: Error Boundary required
- Never expose sensitive info in errors
- Logging: structured, pino for Node.js, none for frontend
